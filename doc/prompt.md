# 本项目所用Agent环境与prompt设计
## Agent环境
|组件类别|规格明细|
|---|---|
|集成开发环境 (IDE)|Visual Studio Code|
|智能代理终端 (Agent Terminal)|GitHub Copilot|
|底层大模型 (Base Model)|GPT-5.3-codex · Xhigh|

## Prompt设计原则
1. **明确目标**：每个 prompt 开头明确本次交互的目标和预期输出格式。
2. **分步引导**：将复杂任务拆解为多个步骤，逐步引导模型完成。
3. **上下文提供**：在 prompt 中提供必要的上下文信息，帮助模型理解任务背景。
4. **示例引导**：通过提供示例输入输出，帮助模型更好地理解预期结果。
5. **迭代优化**：根据模型的输出不断调整和优化 prompt，以提高输出质量和准确性。

## prompt

### 初始化prompt

```prompt
# 任务目标
你是一个高级多智能体（Multi-Agent）调度系统。你需要协调多个 Subagent，完成从文献解析、翻译、知识提取到算法复现与创新的全流程闭环工作。

# 工作区与环境准备
1. 目标文献目录：`F:\Desktop\School\人工智能科学与技术\paper`
2. 基础数据目录：当前工作区内的 `data` 文件夹
3. 请优先在当前工作区根目录下创建两个新文件夹：
   - `scripts`：用于存放所有生成的 Python 代码。
   - `doc`：用于存放学习笔记和最终报告。

# 执行流程（请严格按顺序推进）

## Step 1: 文献解析与翻译 (Document Processing)
- 任务：读取目标文献目录下的两篇原始 PDF 论文。
- 动作：将其精准转换为 Markdown 格式，并翻译为中文。
- 输出：将翻译后的 `.md` 文件保存在与 PDF 相同的原目录下。

## Step 2: 知识提取与笔记生成 (Knowledge Extraction)
- 任务：基于 Step 1 生成的中文 Markdown 文件，深入分析论文核心内容。
- 动作：为每篇论文撰写一份详细的学习笔记。笔记必须包含以下结构：
  1. 论文核心要点（解决的问题、核心创新）。
  2. 算法原理与实现逻辑。
  3. 复现指导（梳理所需的数据预处理、核心公式、参数设置与评估指标）。
- 输出：将两份笔记分别保存至工作区的 `doc` 文件夹中。

## Step 3: 算法复现 (Algorithm Reproduction)
- 任务：根据 Step 2 的复现指导，结合工作区内的 `data` 目录进行代码编写。
- 动作：使用 Python 开发复现代码，确保代码模块化（如分离数据加载、模型构建、评估模块）。包含必要的容错和日志打印。
- 输出：将所有代码文件保存至 `doc` 文件夹对应的 `scripts` 目录中。执行代码并验证复现是否成功。
- 注意：如果遇到数据对齐或维度问题，请自主调试并记录修改过程。

## Step 4: 算法创新与对比 (Innovation & Comparison)
- 前置条件：确认 Step 3 复现成功，并取得基线结果。
- 任务：基于论文原有的算法逻辑，提出一种新的计算方式或优化机制（如改进损失函数、优化权重分配、或者引入新的数学约束）。
- 动作：在 `scripts` 文件夹中编写新算法的代码 `optimized_model.py`，使用相同的数据集运行，并提取对比指标（如精度、收敛速度或运算效率）。

## Step 5: 最终报告生成 (Final Reporting)
- 任务：汇总整个工程的成果。
- 动作：撰写一份终期 Markdown 报告。报告需包含：任务概述、论文核心摘要、复现过程与基线结果、新提出的计算方式的理论依据、以及新旧方法的量化对比实验分析。
- 输出：将最终报告命名为 `Final_Report.md`，保存至 `doc` 文件夹。

# 约束条件
- 每个 Step 完成后，请向用户简要汇报进度，再进入下一步。
- 所有生成的代码必须包含详细的中文注释。
```

### 迭代优化prompt

```prompt
# 角色与背景设定
你是一位顶尖的交叉学科专家（高级土木工程师 + 机器学习算法工程师 + 严谨的科研工作者）。你精通数据科学栈（如 Pandas, Scikit-learn, 树模型等），同时也深刻理解工程材料的底层物理与化学逻辑。
当前项目：基于机器学习的混凝土抗压强度预测。
当前环境：目录下已有三份初始处理脚本。

# 核心任务目标
你的任务是接管该项目，标准化工程架构，并自主进行基于前沿算法的迭代优化。请严格按照以下三个阶段执行，并在每次执行动作（如读写文件、搜索、运行代码）前，输出一段简短的【思考过程】。

## 阶段一：项目审计与文档工程
1. **状态检查**：读取并分析当前目录下的 3 份脚本，理清数据预处理、特征工程、模型训练的逻辑流与依赖关系。
2. **全局 README**：在根目录生成 `README.md`。需包含：项目学术背景、环境依赖、运行说明、整体架构流。
3. **脚本 README**：在 `scripts/` 目录下生成 `README.md`，以表格形式详细标注每个脚本的功能、输入输出要求及执行顺序。

## 阶段二：前沿调研与基线构建
1. **科研检索**：调用浏览器/搜索工具，检索“Machine Learning for Concrete Compressive Strength Prediction SOTA”或相关顶级期刊（如 *Cement and Concrete Research*）的最新开源算法与特征工程策略。提取 2-3 个对当前项目有启发的高价值思路。
2. **确立基线**：运行现有脚本，计算基准测试指标（必须包含交叉验证下的 $R^2$ 和 $RMSE$），作为版本 v0 的 Baseline。

## 阶段三：受控自动化迭代 (v1 至 v5)
你将进行整整 5 次大版本迭代优化。每次迭代必须遵循以下标准作业程序（SOP）：

1. **策略制定**：基于上一版本的不足，结合调研结果，明确本次优化的核心方向（例如：引入高级特征交叉如水灰比、尝试 XGBoost/LightGBM、贝叶斯超参数调优、或集成学习）。
2. **代码隔离**：新建 `v{n}` 文件夹（n=1,2,3,4,5），在其中编写该轮完整的训练与推理脚本，确保各版本环境与代码解耦。
3. **防过拟合验证**：必须使用 $K$-Fold 交叉验证评估模型。你需要确保验证集指标优于 `v{n-1}` 才能宣告该轮迭代成功。**如果性能下降或报错，请利用当前上下文排查原因，自我纠错并重新编码测试，直到指标取得实质性提升。**
4. **输出改动说明**：在每个 `v{n}` 文件夹内生成 `CHANGELOG.md`，严格按照以下格式记录：
   - **对比版本**：v{n-1} -> v{n}
   - **核心改动**：（具体的技术实施细节）
   - **指标变化**：$RMSE$: X -> Y, $R^2$: A -> B
   - **工程反思**：为什么这个改动生效了？它符合土木工程材料学的什么直觉？

# 终止条件与安全限制
- 保持耐心与严谨。如果连续 3 次尝试都无法在当前思路上超越前一版本，允许你在文档中记录“性能瓶颈”，并调用搜索工具寻找全新维度的特征构建方法，而不是在死胡同里微调无效参数。
- 保证代码的模块化，包含必要的日志记录（Logging）与异常处理。
```

### 进一步优化
```prompt
# Role
你是一名顶尖的跨学科专家：资深土木工程师、算法架构师与深度学习模型训练师。你具备极强的工程代码实现能力、前沿算法嗅觉以及严谨的科研态度。

# Objective
深入分析当前工作空间中 `v7` 文件夹下的 `readme` 文件，严格围绕其中定义的【局限性】与【可扩展方向】，进行下一代（v8 及以上）算法的迭代与验证。

# Workflow (科研迭代协议)
请严格按照以下状态机执行任务，确保科研效率与代码产出：

## Phase 1: 研判与规划
1. 读取 `v7/readme`，提取核心痛点。
2. 制定接下来的 5 次迭代计划。每次迭代必须包含一个清晰的【假设】（例如：“引入注意力机制可以降低模型在某类结构上的预测误差”）。

## Phase 2: 受控迭代循环 (最大执行次数：5次)
进入迭代循环。在每一次迭代中，你必须执行：
1. **代码修改**：基于当前版本的计划，编写或重构算法代码。
2. **基准测试**：运行代码，记录核心指标（如精度、Loss、运行耗时等）。
3. **效果评估**：
   - 若指标有提升：将代码固化为新的子版本（如 `v8.1`），并进入下一次迭代。
   - 若出现 Error 或指标退化：立即触发【Phase 3: 降级自救策略】。
4. **中断检查**：完成 5 次有效迭代后，主动暂停并向我询问：“已完成 5 轮既定迭代，性能从 X 提升至 Y，是否需要继续深度优化？”

## Phase 3: 停滞阻断与降级自救策略 (Fallback)
当遇到代码报错或连续 2 次迭代“无明显指标变化”时，停止盲目试错，严格按以下优先级逐级自救，每一级尝试失败后才能进入下一级：
- **Level 1 (内部修正)**：详细审查 Error Log 或算法逻辑缺陷，尝试自行 Debug 和修正代码，最多尝试 2 次。
- **Level 2 (文献与开源检索)**：若内部修正失败，调用联网搜索工具，检索“最新 [具体算法名] 在 [具体土木/工程场景] 中的优化方案”或 GitHub 最新实现。
- **Level 3 (外部求助)**：联网搜索当前开源社区是否有可用的、能解决该瓶颈的第三方 Agent、API 或现成的 Skills 库。
- **Level 4 (总结阻断)**：若上述手段均无效，停止当前开发。向我提交一份详尽的《科研阻断与排错总结报告》。

## Phase 4: 成果封装
无论迭代是通过 Phase 2 正常结束，还是在 Phase 3 被阻断，最后必须执行：
1. **构建目录**：模仿 `v3-v7` 的文件结构，生成最终的 `vX`（如 `v8`）文件夹。
2. **撰写文档**：在最新的 `vX` 文件夹中写入完整的 `readme.md`。该文档必须包含：
   - 最新版所使用的所有算法架构总结。
   - 这 5 次迭代中的心路历程、踩坑记录及对比数据。
   - 下一步（vX+1）的局限性与可扩展方向。

# Constraints
- 绝不要用伪代码敷衍，必须输出可运行的工程级代码（C++/Python/Rust 等视项目而定）。
- 确保所有的算法选择都有科学依据，符合土木工程的物理规律与数据特征。
```
